## What happened

**Increment 33 — `serializer.null.transport`: made the Rust port mimic Java's null-omission
serialization (a faithfulness parity fix).** Committed `75bf1ba`, pushed to `acn-ericlaw/mercury` main.

- **Trigger (field test).** The maintainer tested the Java `/api/companion/{id}/sync` and the success
  response omitted `error` (null). Java does this by design: `serializer.null.transport` (default
  `false`) makes **both** wire serializers drop nulls — Gson omits null map/object fields unless
  `serializeNulls()` (`SimpleMapper`), and `MsgPack.packMap` skips null map values
  (`if (supportNulls || value != null)`). The Rust port did the **opposite** (always transported nulls
  via serde), so it would emit `"error": null`. Maintainer's call: **full parity (JSON + MsgPack)**.
- **Fix.** New `crates/platform-core/src/serializer.rs`: `null_transport()` (cached read, default
  false) + `strip_nulls` / `strip_nulls_always` (recursively drop `Nil` **map** entries; **array**
  slots preserved — matches Gson field-omission and Java `packList`). Applied at every wire boundary:
  JSON HTTP response (`server.rs` `envelope_payload`), WS text/tee (`ws_server.rs`), outbound HTTP
  body (`http_client.rs`), and the MsgPack encoder (`envelope.rs::to_bytes`, guarded so a scalar body
  or transport-on path encodes with no extra clone).
- **Result.** Default now matches Java byte-for-byte on null handling (the `/sync` success response
  omits `error`); `serializer.null.transport: true` restores explicit null transport. Documented in
  the minigraph-playground `application.yml`; recorded as increment 33 in `docs/INCREMENTS.md`.
- **Tests.** 5 serializer unit tests (map-drop, nested recursion, array-slot preservation,
  maps-in-arrays, scalar pass-through) + `playground.rs` `/sync` round-trip now asserts `error` is
  omitted on success. Full workspace green **except** the known pre-existing `graph_runtime` boot flake
  (two `#[tokio::test]`s per binary — reproduces on HEAD; each passes in isolation). event-script (the
  msgpack inter-function transport path) fully green. fmt + clippy clean.
- Java side unchanged — it is the source of truth this mirrors.

**Also (same session, `0f223be`):** maintainer gave the rationale + invariants — nulls are omitted
because uninitialized PoJo fields serialize as noise; `serializer.null.transport=true` exists for apps
that must tell "key present with null" from "key absent". Locked the invariants in the module doc,
config comment, and INCREMENTS, with two added tests: **(1)** affects map key-values only; **(2)**
array elements (incl. `Nil`) are always kept so ordering holds; **(3)** an empty `[]`/`{}` is a real
value, never null. My implementation already satisfied all three (only `Value::Nil` map entries are
filtered) — the clarification is now documented + test-locked.

**Perf follow-up (same session, `3a53fbb`).** Maintainer asked whether the null feature adds
in-memory-transport overhead. Traced it: `EventEnvelope::to_bytes()` (the only serialize) is called
**only** by `spill()` (platform.rs) on the ElasticQueue back-pressure path — the in-memory fast path
moves the envelope over a tokio channel unserialized, so **zero** overhead there. On the spill path
the clone+strip is now gated by `has_nil_map_entry` (read-only, allocation-free, early-return; a `Nil`
array element doesn't count) — so the common case (scalar body, or structured body with no null map
values, or transport on) pays only an O(n) scan and encodes `self` directly (no clone, no rebuild).
Same guard applied to `strip_nulls` for the JSON edges. Design rule captured in
`[[serializer-null-transport-invariants]]`.
