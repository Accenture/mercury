## What happened

**HTTP-boundary content-type dispatch now mirrors Java exactly (maintainer-directed parity
fix).** The maintainer's manual `/sync` test noticed the Rust port accepted a plain-text
command mislabeled `Content-Type: application/json` and asked how both engines handle
content types.

- **Investigation (code + wire-verified):** the current Java (v4.8.6) `HttpRouter` is
  *tolerant* on that exact case ‚Äî under `application/json` it only attempts the serializer
  when the trimmed body is bracket-wrapped (`{‚Ä¶}`/`[‚Ä¶]`), any parse failure falls back to raw
  text ‚Äî so both engines already agreed there. But three real divergences existed: (1) Rust
  *sniffed* JSON-looking bodies under any content type (Java never parses outside
  `application/json`); (2) a **missing/unknown content type** in Java yields a **byte[]**
  body (`handleBinaryContent`) while Rust text-decoded it; (3) an empty `application/json`
  body is `{}` in Java, was null in Rust. A raw-socket wire capture of the Java client
  (scratch probe, deleted after) settled a key fact: **the Java client sends no default
  content-type** (fetcher-style Map POSTs go out header-less + chunked) ‚Äî the canonical
  POST-provider fixtures work because they explicitly map
  `text(application/json) -> header.content-type` (tutorial-6, both repos). The Rust
  fixtures + the AI grammar's POST example already carry that line, so nothing
  grammar-conformant relied on the sniffing (webapp POSTs all declare JSON too).
- **Fix (maintainer decision: mirror exactly):** `parse_body` in
  `crates/platform-core/src/automation/server.rs` now reproduces Java `handlePayload`:
  `application/json` (bracket-guard, text fallback, empty ‚Üí `{}`), `application/xml` ‚Üí raw
  text (XML parse stays deferred, matching the client's response side),
  `application/x-www-form-urlencoded` (exact) ‚Üí fields into `parameters.query` + null body,
  `text/html`/`text/plain` ‚Üí raw text, anything else incl. missing ‚Üí **MsgPack binary**
  (empty ‚Üí null). Content-type matched on the `;charset`-stripped value, case-sensitively
  like Java. Binary bodies ride as `rmpv::Value::Binary` substituted into the JSON-shaped
  event map in `build_event` (serde_json can't carry bytes).
- **Tests:** rewritten `body_parsing` unit test (all dispatch rules) + new end-to-end
  `body_dispatch_mirrors_java_content_type_rules` in `tests/rest_automation.rs` with a
  `BodyProbe` function reporting the body *kind* that reaches a function (map / text /
  bytes / null / query-merge). Workspace: 202 tests, 0 failures; clippy 0; fmt clean.
- **Docs:** D10 dispatch section in `docs/design/platform-core-port.md` updated with the
  exact rules + the no-default-content-type wire finding.

## Decisions

- **Mirror Java exactly at the HTTP boundary** (maintainer, 2026-07-19) ‚Äî the earlier
  lenient Rust behavior (JSON sniffing, no binary path) is gone; a caller that wants a
  parsed map MUST declare `application/json`, exactly as in Java. The one remaining
  deliberate divergence is `application/xml` arriving as raw text instead of a parsed map
  (no XML parser in the Rust stack; same deferral as the HTTP client's response side) ‚Äî
  documented in code + D10.
