### Inspect endpoint — composite scalar keys (both repos, parity)
- The companion read run output via `GET /api/inspect/{id}/{key}`. `…/output` (a Map) → 200,
  but `…/output.body` (a scalar) → **404**. Investigated + compared with Java: **identical
  behavior** — both `download_content`/`downloadContent` resolve the composite key through the
  MultiLevelMap (`get_element`/`getElement`), but the handler only returned Map/List and 404'd
  scalars. So it was faithful parity, *not* a Rust divergence.
- Maintainer chose to **enhance both + keep parity**, using the shape the `inspect {key}`
  console command already emits: **wrap the resolved value in `{inspect, outcome}`** for ANY
  type (scalar / primitive / Map / List all serialize as clean JSON), 404 only when the key
  resolves to nothing. This also makes the endpoint consistent with the command's inline output.
- **Rust:** `rest.rs` handler + `tests/playground.rs` (asserts `…/output.body` →
  `{inspect:"output.body", outcome:"hello world"}`) + `tests/resources/rest.yaml` route
