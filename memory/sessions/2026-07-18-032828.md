## What happened

**Bug fix (both repos, parity) — a Dictionary node's `input[]` is not a data mapping.**
Surfaced by the AI-companion validation of tutorial-3: `create node person-name … input[]=person_id`
(a Dictionary) was rejected with `Invalid 'input' entry - syntax must be 'LHS -> RHS'`.

- **Root cause (Java, mirrored in Rust):** the create/update-node console command runs every
  `input[]`/`output[]`/`mapping[]`/`for_each[]` entry through the `LHS -> RHS` validator
  (`convertMappingProperties`/`convertOneMappingLine`, added with the simple-type-matching
  deprecation, Java #130). But a **Dictionary** node's `input[]` holds an input *parameter* with an
  optional **colon-default** (`person_id` or `person_id:100`), not a mapping — confirmed by the
  runtime consumer `GraphApiFetcher.java:298-324` (colon-split, no arrow). So deployed graphs
  **execute** fine (the maintainer's Java run showed tutorial-3 fetching `Peter / 100 World Blvd`);
  only the console command was broken. The Rust port had faithfully copied the Java bug.
- **Fix (Java first, then Rust — maintainer's call):** make the validator **node-type-aware** —
  a Dictionary's `input[]` is exempt from the mapping-syntax check (stored as-is); Provider/Fetcher
  `input[]` and all `output`/`mapping`/`for_each` still validate. Reordered the handlers so the node
  type is known before validation.
