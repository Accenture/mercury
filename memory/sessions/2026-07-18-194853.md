## What happened

**Fixed the companion `/sync` drain so its `run` response is complete and byte-identical across the
Rust and Java engines — the AI-companion REST surface is language-neutral.**

- **Root cause (not what I'd noted before).** My prior session's "cross-port note" — that Java `run`
  emits only the traversal trace while Rust also echoes output.body — was **wrong**. The maintainer's
  screenshot proved the Java WS console already echoes `Executed … / {output:{…}} / Graph traversal
  completed in N ms`, identical to Rust. The real defect was the **`/sync` endpoint's drain**: `run` is
  the only *asynchronous* command — the command handler launches the traveler and replies immediately,
  then the traveler streams `Walk to…` / `Executed…` / the `output` map / a terminal line to `out`
  **after** the reply. The FIFO sentinel enqueued right after the reply raced (and usually beat) that
  async tail, so `/sync run` was truncated to just `Walk to root` / `Walk to end`.
- **Fix (both engines, symmetric).** Drain a traversal on the traveler's **terminal line** —
  `Graph traversal completed in N ms` (success) or `Graph traversal aborted` (failure), always emitted
  last, so once seen every prior line is buffered. Synchronous commands keep the FIFO sentinel. The
  bounded poll/await is only a safety net; the signal drives correctness (the maintainer's ask: "a
  clean end-of-transmission signal from the engine, not a timeout").
- **Made the terminal a contract.** Several failure paths (no instance yet, missing root/end, unknown
  skill, dangling next-node) ended with only a *reason* line. They now emit the reason **then** the
  canonical `Graph traversal aborted` (`emit_aborted` / `emitAborted`), so a companion mistake such as
  `run` before `instantiate` returns promptly (`ok:false`) instead of hanging to the timeout.
