## What happened

Maintainer raised a design question: Java's `EventEmitter.sendWithEventBus()` runs
`event.script.manager` and `task.executor` **directly** (fresh virtual thread,
bypassing the Vert.x bus + ServiceQueue) so the engine behaves as part of the event
core — do we need this in Rust? **My analysis (accepted):** port it, but for different
reasons than Java —

1. Java's biggest win (skipping `toBytes()` serialization per hop) **does not exist
   in Rust**: our bus is zero-copy (owned envelopes through channels).
2. What DOES transfer: **concurrency** (my E-4 one-instance registration serialized
   ALL orchestration across ALL flows — a real bottleneck this question caught) and
   **liveness** (an event router feeding itself through bounded mailboxes risks
   circular waits under saturation; Java dodges it by construction).
3. Bonus: tracing parity — Java's direct path bypasses `WorkerHandler`, so the engine
   is never trace-bracketed (my E-4 note "Java traces its engine" was wrong in
   practice); only the flow-summary span is emitted.

**Maintainer refinement:** agreed except the proposed `direct` flag on `#[preload]` —
**hide the optimization from macros and interface contracts** so developers cannot
abuse it (application functions must stay on the reactive back-pressure path).

**Shipped (repo increment 15):** a **private** `RESERVED_ENGINE_ROUTES` list in
platform-core (`event.script.manager`, `task.executor` — the same names Java
hard-codes) checked by `Platform::deliver` AND the worker reply path; matches run via
`spawn_direct` (fresh tokio task, instance 1, failures logged — the
`runTaskExecutor` analog). `RouteEntry` now carries the function Arc (private struct,
no API change). No macro flag, no `FunctionOptions` field — invisible to users.

**Proof tests** (`tests/direct_execution.rs`): 10×150 ms events per reserved route
with ONE worker instance → peak concurrency > 1, batch finishes in a fraction of the
serialized time; an identical probe on a normal route → peak EXACTLY 1 (the control
proving back-pressure is intact elsewhere). Plus 20 simultaneous end-to-end greetings
flows (distinct cids + state machines) in flow_runtime. **148 workspace tests, 0
failures; clippy 0; fmt clean.** ES design §5d amended (serialization claim
superseded) + new §5d.1; INCREMENTS row + section 15.

## Decisions

- Hidden-by-design (maintainer): reserved names in the platform, not a capability in
  the API — mirror of Java's approach, documented as "an engine privilege, not an API".
- Safety argument recorded: stateless routers; per-callback scratch mapping under one
  continuous dataset lock (E-4 design) makes concurrent callbacks correct; user
  functions keep bounded mailboxes; flows are TTL-bounded.
