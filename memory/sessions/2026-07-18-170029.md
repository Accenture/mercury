## What happened

**Addressed the AI-companion tooling feature gap surfaced by the Tutorial-4 exercise: designed +
prototyped a synchronous, structured companion response.** (Also tore down the WS subscriber from the

- **The gap (maintainer's framing, confirmed):** the companion HTTP surface is a *write-only command
  bus* — `POST /api/companion/{id}` returns `{status:"accepted"}` and the real outcome/errors stream
  WS-only, so an AI caller is blind (Tut-4: HTTP 200 while the run had aborted) and must poll
  graph-shape/inspect to guess. A "true AI companion" needs **synchronous, self-describing feedback**.

- **Design note:** `docs/design/ai-companion-sync.md` — an **additive** `POST /api/companion/{id}/command`
  returning `{ok, output, error, result}`: errors in-band, run/inspect results folded in; the existing
  fire-and-forget endpoint and the WS console are unchanged. Mechanism reuses `PostOffice::request`
  (RPC) + a private capture route (`Platform::register`). Considered + deferred: MCP tool server;
  handler-returns-transcript refactor. Maintainer decisions (via AskUserQuestion): **design note +
  Rust prototype**, **additive sync mode**.

- **Blueprint thread:** `bp-companion-sync` (serves `vision-mercury`) — plan: prototype in Rust →
  re-verify with the Tut-4 companion → PR the design to the Java upstream (the gap exists there too).

- **Prototype (Rust R&D, built + tested):**
  - `rest::post_companion_command_sync` + a `CaptureSink` composable function. It mints a private
    `companion.sync.{uuid}` route, registers a 1-instance capture sink, RPCs the command to
    `graph.command.singleton` with that route as `out`, then enqueues a FIFO sentinel to know the
    buffer is fully drained (deterministic — `say()` is fire-and-forget), drains, releases the route,
    and returns the structured envelope. `ok` is derived heuristically (no `ERROR:`/`aborted`/… line);
    the design note flags a cleaner handler-returned status as a follow-up.
  - Dev-gated wrapper `PostCompanionCommandSync` (`post.companion.command.sync`, `#[optional_service`
    `("app.env=dev")]`) in `lib.rs`; rest.yaml route `/api/companion/{id}/command` in the example app.
  - **Verified:** new integration test `companion_sync_returns_outcome_in_band` — invalid command →
    `ok:false` with the error **in-band** (the Tut-4 blind spot, closed); valid command → `ok:true`,
    `error:null`, `output` populated. Full `cargo test --workspace` green, clippy + fmt clean.
