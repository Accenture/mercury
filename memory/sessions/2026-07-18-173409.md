## What happened

**Refined the synchronous companion endpoint to tee its output to the WebSocket console —
`bp-companion-sync` now delivers real-time human+AI collaboration.** From a maintainer observation:
after the sync prototype, a human watching the Playground UI saw a *quiet* console even as the model
changed (they had to edit the root node to force a re-render), because sync mode captured output to
the caller only.

- **Refinement (maintainer's suggestion):** the private `companion.sync.{uuid}` `CaptureSink` now
  **fire-and-forget forwards** each captured line (except the internal sentinel) to the session's real
  WebSocket `.out` route, in addition to buffering it for the in-band response. So the AI caller and a
  watching human see the **same output live**.
- **Multi-party, for free:** the command service's `handle_command` already re-dispatches a primary's
  command to each subscriber's own `.out`, so any `session subscribe`d watcher (e.g. a product owner
  on the UI) also sees it. Combined with the tee, this realizes the headline goal — an architect + AI
  draft a graph on one live session while others watch/participate; suspend/resume across sprints via
  export/import.
- **Deterministically tested:** extended `companion_sync_returns_outcome_in_band` with an `OutTap`
  stand-in registered at the session's `.out` route; asserts the sync command's output (`node root
  created`) is teed there — not just returned to the caller. Test green, clippy + fmt clean.
- **Docs:** `docs/design/ai-companion-sync.md` updated (tee promoted from "out of scope for v1" to a
  v1 feature, with the collaboration rationale); the endpoint doc comment updated.

**Remaining on `bp-companion-sync`:** (a) PR the design to the Java upstream; (b) document the sync
endpoint in the ai-agent-guide; (c) a live multi-party demo (architect+AI draft, PO subscribes and
watches) — needs the server restarted on the new binary.
